# 產品需求文件 (PRD)

## 專案名稱
校園設施體感地圖

## 專案描述
建立一個輕量化的即時回報平台，透過「三秒極簡回報」機制，讓學生互助分享校園空間真實的感官狀態（如冷熱、擁擠程度），解決現有系統僅提供靜態數據而缺乏即時體感資訊的問題。

# 產品需求文件 (PRD)：校園設施體感地圖專案規劃

## 1. 專案概述

### 背景與動機
在校園生活中，學生常面臨公共空間資訊不透明的困擾。例如：圖書館顯示有空位，到現場才發現冷氣故障悶熱；或是想去買午餐，卻無法得知即時的人潮擁擠程度。現有的校園系統多半只提供靜態數據（如座位總數），缺乏「感官性質」的即時回饋。我們希望建立一個輕量化的回報平台，讓學生互助分享空間的真實狀態。

### 目標用戶
- **大學生**：主要使用者，需要尋找舒適讀書環境或避開排隊人潮。
- **教職員**：查看行政空間或公共區域的即時使用狀況。
- **學校管理方**：透過數據統計了解哪些區域長期處於「過熱」或「空調不足」的狀態。

### 核心價值主張
極大化降低資訊分享的門檻，透過免打字、一鍵點擊標籤的方式，結合自動定位與時間抓取，打造出具備高即時性、高準確度的校園空間動態狀態地圖。

- **大學生**：主要使用者，需要尋找舒適讀書環境、避開排隊人潮，以節省時間。
- **教職員**：查看行政空間或公共區域的即時使用狀況。
- **學校管理方**：透過數據統計了解哪些區域長期處於「過熱」、「空調不足」或設備常態故障的狀態，以優化資源配置。

### 核心價值主張
聚焦於「校園設備即時動態」，提供針對影印機、飲水機、公用電腦等熱門設備的「排隊中、可用、維修中」群眾標註功能，讓校園資訊由靜態轉為即時、動態互助的體驗。

## 2. 功能需求

### 1. 三秒極簡回報
作為 **大學生**，我希望 **能在三秒內點擊「超冷、悶熱、爆滿、空曠」等體感標籤進行回報且無需輸入文字**，以便 **快速分享當下環境狀態且不耽誤時間**。

### 2. 自動定位與時間記錄
作為 **大學生**，我希望 **系統能自動抓取我回報時的時間與所在位置**，以便 **省去手動選擇地點的麻煩，確保回報資訊的效率與準確性**。

### 3. 即時體感地圖與狀態列表
作為 **大學生**，我希望 **能在地圖或列表上直觀地看到各個空間目前的體感狀態與擁擠度**，以便 **決定是否要前往該地點**。

### 4. 狀態時效與可信度機制
作為 **大學生**，我希望 **能看到該地點回報資訊的時效性（例如：5分鐘前回報）**，以便 **判斷該資訊是否仍然符合當下狀況**。

### 5. 數據統計與管理儀表板
作為 **學校管理方**，我希望 **能檢視各地點的歷史回報數據統計（如某教室被頻繁回報為悶熱）**，以便 **發現設備潛在問題並安排維修或改善**。
即使整體範圍聚焦於設備動態，我們仍可將功能拆分成以下幾個核心故事：

1. **設備狀態查看**
   - 作為 **大學生**，我希望 **能在清單上看到各項設備（如影印機、飲水機）的即時狀態（排隊中/可用/維修中）**，以便 **決定是否前往該地點使用**。
2. **快速群眾標註**
   - 作為 **大學生**，我希望 **能在到達現場時，快速點擊按鈕回報設備的最新狀態**，以便 **讓其他同學得知真實情況**。
3. **設備分類篩選**
   - 作為 **大學生/教職員**，我希望 **能依照設備類型（如：只看飲水機，或只看公用電腦）進行篩選**，以便 **快速找到符合我當下需求的設備**。
4. **狀態統計與匯總**
   - 作為 **學校管理方**，我希望 **能有一個簡單的後台或數據檢視介面，看到哪些設備被標註「維修中」最多次、或是哪裡的設備總是「排隊中」**，以便 **進行設備的汰換評估或維修派工**。
5. **標註時效性機制**
   - 作為 **系統**，我希望 **使用者的標註狀態具有時效性（例如 30 分鐘或 1 小時後自動重置為預設狀態）**，以便 **確保畫面上顯示的狀態是最新的，避免過期的「排隊中」誤導其他學生**。

## 3. 非功能需求

- **技術限制**：
  - 後端：Flask
  - 前端：Jinja2 (可搭配原生 HTML / CSS / JS，如使用 Geolocation API 取得定位)
  - 資料庫：SQLite
- **效能考量**：系統需能承受特定尖峰時段（如下課、午餐時間）的集中回報，輕量化 API 確保快速回應與寫入。
- **安全與防呆考量**：
  - 基本防護：避免同一用戶在短時間內針對同一地點進行惡意洗版（Rate Limiting）。
  - 身分識別：需有輕量級的用戶識別機制（如 Session、Cookie 或簡單註冊/登入）。

## 4. MVP 範圍

### Must Have (必須有)
- 極簡體感回報介面，提供預設標籤（超冷、悶熱、爆滿、空曠 等）。
- 支援瀏覽器 Geolocation API 自動抓取地點（或提供常見地點快速下拉選單），並記錄回報時間。
- 即時狀態顯示頁面（列表式或簡單地圖呈現各區域最新狀態）。

### Should Have (應該有)
- 資訊時效性顯示（標示回報時間，過期的狀態視覺上變淡或自動隱藏）。
- 簡單的防刷機制（同一裝置/帳號短時間內不得重複回報同一地點）。

### Nice to Have (可以有)
- 提供給學校管理方的長期數據儀表板與圓餅圖/長條圖分析。
- 個人回報紀錄與校園回報貢獻排行榜（鼓勵學生積極使用）。
  - 系統架構將使用 Flask（Python）作為後端框架。
  - 前端使用 Jinja2 模板引擎進行渲染，搭配 Vanilla CSS/JS。
  - 資料庫採用 SQLite 進行輕量化資料儲存。
- **效能考量**：
  - 由於多數情境為學生在校園內移動時使用，介面需具備良好的行動裝置適應性（Mobile First），並確保載入速度夠快。
- **安全與防護考量**：
  - 需要基本的防護機制（例如簡易的 IP 限制或回報頻率限制），防止惡意點擊或洗版干擾標註真實性。
